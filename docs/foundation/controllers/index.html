Controllers

When using Vivid, each Controller that is going to serve a Feature needs to extend the Vivid\Foundation\Http\Controller base class. This base class has two main functions.

The first one is providing access to the ServesFeaturesTrait which in turn allows you to use the $this->serve() method. This is mandatory in order to serve Vivid Features.

Secondly, the constructor of the Vivid Controller is responsible for adding the Controller class FQN to the Vivid\Foundation\Instance for use in debugging.

function __construct()
{
    resolve('Vivid\Foundation\Instance')->setController(
        get_class($this)
    );
}

Because this action is done inside the constructor, each subsequent controller that extends the base Vivid controller needs to call parent::construct() if it implements a custom constructor.

If for whatever reason you forgo doing that, the Vivid\Foundation\Instance::$controller variable will stay null and consequently Vivid\Foundation\Instance::getController() will return null.

Generating Controllers

In order to generate a Controller using the Vivid Console type the following command:

./vendor/bin/vivid make:controller <controller> <device>

By default this command will generate a very basic controller class that looks like this:

<?php

namespace App\Devices\ExampleDevice\Http\Controllers;

use Illuminate\Http\Request;
use Vivid\Foundation\Http\Controller;

class ExampleController extends Controller
{

}

If you want to save time and generate a resource controller you can do it by using the --resource option. This will generate a controller that implements the following functions: index, create, store, show, edit, update, destroy.

Because a controller only serves ONE feature and should do nothing else, you have controllers with a single line of code inside.
