Devices

Think of Devies as an I/O device. The content of Devices is similar to the default Laravel structure that we are used to seeing. Each one of them is responsible for providing an interface with out application. They have their own set of controllers, routes and features.

Each Device must be independed for the others and it should be able to function on its own without the existance of the other devices.

It is acceptable for devices to use the classes located outside of the /Devices.

Generating a Device

In order to generate a Device using the Vivid Console type the following command:

./vendor/bin/vivid make:device ExampleDevice

Directory Structure

/app
    /Devices
        /ExampleDevice
            /Console
            /database
            /Features
            /Http
            /Providers
            /resources
            /routes
            /Tests

Routing

Each Device gets it's own set our routes. When generating a Device, the routes will be registered automatically by the RouteServiceProvider located at /app/Devices/<device>/Providers/RouteServiceProvider.

By default, you will have a web.php and an api.php file which you will use accordingly. The routes inside these will be prefixed with /api/<device>/ and /web/<device/>.

Here is an preview of the RouteServiceProvider:

class RouteServiceProvider extends ServiceProvider
{
    /**
     * Read the routes from the "api.php" and "web.php" files of this device
     *
     * @param \Illuminate\Routing\Router $router
     */
    public function map(Router $router)
    {
        $namespace = 'App\Devices\ExampleDevice\Http\Controllers';
        $pathApi = __DIR__.'/../routes/api.php';
        $pathWeb = __DIR__.'/../routes/web.php';

        $this->loadRoutesFiles($router, $namespace, $pathApi, $pathWeb);
    }
}

Examples

Lets consider the following scenario:

You are building a small ERP which needs to have an API for exposing the data and interacting with the app, a Web insterface and an intergration with few POS systems.

Using the Vivid Architecture you should create a Device for each endpoint of interaction with your app. More specifically, you should make a Device for the Web, the API and for each POS that demands to communication with your app. It's easy to assume that the vast majority of apps will only ever need a Web and an API device and that actually correct, however the example with the ERP illustrates how devices can help you even further when it comes to seperating concerns.

Each POS vendor migth have different ways of fetching of sending data to external services. The I/O operations should be distinct and isolated.

At all times you should be able to just delete a device folder the the app should be functional (without the endpoints which were contained inside the now deleted device, of curse)
